Why Proof Writing Matters in CS

Writing proofs is like writing documentation for your logical reasoning. Just as code needs structure (functions, modules, classes), proofs need structure (definitions, claims, steps).

The Code Connection

In programming, you don't just write print(x) and hope it works‚Äîyou define your variables and logic first. A proof is the same: it's "Documentation" for your thinking!
The Parallel:
  • Definitions = Type declarations and interfaces
  • Claims = Function signatures (the promise)
  • Proof = Implementation (the logic)
üìù Simple: Writing a proof is just telling a story to a skeptical friend. You can't just say "it works." You have to say "First we did this, which forced this to happen, so finally this must be true." If your friend asks "Why?" and you can't answer, the proof is broken.

Section 1

The Standard Proof Structure

The Standard Format

Every proof follows this 4-step structure:
  1. Definitions: Define your terms precisely.
  2. Claim: State exactly what you intend to prove.
  3. Proof: The step-by-step logical argument.
  4. Conclusion (QED): Signal completion with ‚àé.
Pro Tip: "Quod Erat Demonstrandum" (QED) is Latin for "Which was to be demonstrated." In Simple Mode: "See? Told ya!"
Section 2

Writing Good Definitions

A Definition is like an ID card for a concept. It should tell you exactly who they are without any guessing.

Pattern: Existence vs. Universal

  • Existential: "n is even if there exists an integer k such that n = 2k."
  • Universal: "A room is Empty if everyone is gone."
Tip: Don't try to prove a definition. It's just a name we agreed on. You don't prove that "Blue" is "Blue"‚Äîyou just use the word!

Section 3

Stating Claims and Theorems

In math, we have different names for "Truths" depending on how big they are.

The Hierarchy

  • Theorem: The "Big Boss." The main result you are proving.
  • Lemma: The "Side-kick." A helper result used to prove bigger things.
  • Corollary: The "Freebie." Something that becomes obviously true once you win the Big Boss.
Key Point: A good claim is like a good function name. It should say exactly what's going in (If...) and exactly what's coming out (Then...).
Section 4

Writing the Proof

A proof is a story, not just a list of symbols. Talk to the reader!

The "Traffic Signs" of Math

  • "Let..." means "I'm introducing a new variable."
  • "Since..." means "Here's the evidence for my next step."
  • "Therefore..." means "This is the logical conclusion."
The "Friend" Rule: If a peer asks "Why?" at any point, you need more detail. Don't skip logical steps, even if the math seems simple.

Complete worked example

Theorem
For all integers n, if n¬≤ is odd, then n is odd.
Proof (by contrapositive)
We prove the contrapositive: if n is even, then n¬≤ is even.

Let n be an even integer.
By definition, n = 2k for some integer k.
Then n¬≤ = (2k)¬≤ = 4k¬≤ = 2(2k¬≤).
Since 2k¬≤ is an integer, n¬≤ has the form 2m where m = 2k¬≤.
Therefore n¬≤ is even by definition.

Since the contrapositive is true, the original statement holds.

Section 5

The Mathematician's Style Guide

Common Logical Traps

  • The "Mystery Guest": Don't use a variable (like $x$) if you haven't told us what it is!
  • The "Because I Said So": Don't skip steps. Every logical jump needs a reason (e.g., "by definition").
  • Proof by Example: Showing it works for the number 5 doesn't prove it works for every number.
Golden Rule: If you read your proof out loud, it should sound like a clear story, not a secret code.
