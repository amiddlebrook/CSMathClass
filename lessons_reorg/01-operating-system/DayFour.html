Why Proofs Matter in CS

Every algorithm correctness argument is a proof. When you claim your sorting function works, you're implicitly proving it. Understanding proof techniques makes you better at reasoning about code.

Key insight: Debugging is just proof-reading. You're checking if your implicit proof of correctness actually holds.
üïµÔ∏è Simple: Think of a proof like a chain of iron links. If the first link is solid (your assumptions), and every connection to the next link is welded tight (valid logic), then the final weight (the conclusion) will definitely hold.

Where you'll use this

  • Algorithm Analysis: Proving correctness, termination, complexity bounds
  • Loop Invariants: Showing loops do what they claim
  • Recursive Functions: Induction proves recursion is correct
  • Security: Cryptographic security proofs
  • Type Systems: Type safety proofs guarantee no runtime type errors
PythonEvery algorithm has an implicit proof
def binary_search(arr, target):
    """
    Implicit proof claims:
    1. If target exists, we find it (correctness)
    2. Loop terminates (lo eventually meets hi)
    3. O(log n) time (halving search space)
    
    Each claim needs a proof technique!
    """
    lo, hi = 0, len(arr) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if arr[mid] == target: return mid
        elif arr[mid] < target: lo = mid + 1
        else: hi = mid - 1
    return -1
Section 1

Direct Proof

A Direct Proof is the most straightforward way to prove something. You start with what you know and follow a chain of "If this, then that" until you reach your goal.

The Method (The Recipe)

To prove P ‚Üí Q: Assume P is true, apply definitions, and show Q must follow.
Analogy: Think of it like a train. Each logical step is a segment of track. If every segment is connected correctly, the train flows perfectly from start to finish.

Example: Sum of two even numbers

Claim: Even + Even = Even. Proof: 1. Let a and b be even. By definition, a = 2m, b = 2n. 2. a + b = 2m + 2n = 2(m + n). 3. Since (m + n) is an integer, the result is of the form 2k. 4. Therefore the sum is even. ‚àé
Section 2

Proof by Contrapositive

Sometimes proving "If A, then B" is hard. But proving "If NOT B, then NOT A" (the Contrapositive) is exactly the same thing.

The "Switcheroo"

Analogy: "If it's a dog, it has four legs."
Contrapositive: "If it DOES NOT have four legs, it IS NOT a dog."
If the second one is true, the first one must be too!

Example: If n¬≤ is even, then n is even

Prove contrapositive: If n is odd, then n¬≤ is odd. Proof: 1. Assume n is odd: n = 2k + 1. 2. n¬≤ = (2k + 1)¬≤ = 4k¬≤ + 4k + 1 = 2(2k¬≤ + 2k) + 1. 3. This is clearly odd. 4. Since the contrapositive is true, the original claim is true. ‚àé
Section 3

Proof by Contradiction

Proof by Contradiction is like an argument where you say, "Okay, let's pretend you're right for a second. If you're right, then something impossible happens. Since impossible things can't happen, you MUST be wrong!"

The "Nuh-uh" Method

Strategy: Assume the opposite of your claim. Show it leads to nonsense (e.g. 1 = 0).
Example: Prove there is no largest integer. 1. Assume there IS a largest integer L. 2. But L+1 is also an integer and clearly larger than L. 3. This contradicts the claim that L was the largest. 4. Therefore, no largest integer exists. ‚àé
Section 4

Mathematical Induction

Induction is the "Domino Effect" proof. It proves a claim for 1, then shows if it works for $k$, it must work for $k+1$. This covers every number in the line.

The Two Steps

1. Base Case: Prove it works for the start (usually $n=1$).
2. Inductive Step: Show that $P(k) \implies P(k+1)$.
Analogy: If the first domino falls, and every falling domino knocks over the next one, then EVERY domino will fall no matter how long the line is!

Example: Sum of first $n$ integers

Formula: 1 + 2 + ... + n = n(n+1)/2. 1. Base (n=1): 1 = 1(2)/2. ‚úì 2. Assume n=k: sum = k(k+1)/2. 3. For k+1: sum = [k(k+1)/2] + (k+1) = (k+1)(k/2 + 1) = (k+1)(k+2)/2. ‚úì It works for all $n$!
Section 5

Loop Invariants

A Loop Invariant is a condition that stays true before and after each iteration of a loop. It's the "rule" your algorithm never breaks.

The Correctness Recipe

1. Initialization: True before loop starts.
2. Maintenance: If true before a step, it's true after.
3. Termination: When loop ends, Invariant + Exit Condition = Correctness.
Analogy: Sorting a deck of cards. The "rule" is: "The hand I'm holding is always sorted." No matter if you've picked up 1 card or 50, that rule stays true!

Example: Binary Search

Invariant: "If the target exists, it is in the range [lo, hi]." Maintenance: - If mid < target, target must be in [mid+1, hi]. - If mid> target, target must be in [lo, mid-1]. Both cases keep the target inside our shrinking window! ‚àé
