Section 1

IEEE 754 Standard

Real numbers are infinite. Computers are finite. To bridge this gap, we use scientific notation in binary: $\pm 1.mantissa \times 2^{exponent}$.

The Bit Layout

Value = (-1)^S \times (1 + M) \times 2^{E - Bias}

Single Precision (32-bit)

  • Sign (S): 1 bit
  • Exponent (E): 8 bits (Bias 127)
  • Mantissa (M): 23 bits
  • Precision: ~7.2 decimal digits

Double Precision (64-bit)

  • Sign (S): 1 bit
  • Exponent (E): 11 bits (Bias 1023)
  • Mantissa (M): 52 bits
  • Precision: ~15.9 decimal digits
Implicit Leading Bit: We don't store the "1" before the decimal point. It is assumed. This buys us one extra bit of precision.
Section 2

Special Values

The standard reserves certain exponent patterns for non-numbers.

Zero (0.0, -0.0): Exponent = 0, Mantissa = 0. Note that signed zero exists! `1.0 / -0.0 = -Infinity`.
Infinity (+inf, -inf): Exponent = All 1s, Mantissa = 0. Result of overflow or division by zero. `inf + 5 = inf`.
NaN (Not a Number): Exponent = All 1s, Mantissa ≠ 0. Result of `0/0`, `inf - inf`, `sqrt(-1)`. Warning: `NaN != NaN` is true.
Subnormal Numbers: Exponent = 0, Mantissa ≠ 0. Allows "gradual underflow" near zero, but fewer significant digits.
Section 3

Sources of Error

1. Rounding Error

Most real numbers (like 0.1) have infinite repeating binary representations. Truncating them creates an error up to Machine Epsilon ($\epsilon_{mach}$).

0.1 + 0.2 = 0.30000000000000004 (Because 0.1 is 0.0001100110011...)

2. Catastrophic Cancellation

Critical: Subtracting two nearly equal float numbers destroys precision. If $a \approx b$, then $a-b$ leaves only the "garbage" noise bits at the end.
Example: Quadratic Formula x = \frac{-b + \sqrt{b^2 - 4ac}}{2a} If b^2 >> 4ac, then \sqrt{b^2 - 4ac} \approx b. Numerator becomes -b + b (cancellation!). Fix: Rationalize the numerator or use an alternate form.
Section 4

Numerically Stable Algorithms

Summation

Adding $10^6$ small numbers to a large number one by one causes precision loss.

Kahan Summation Algorithm: Keeps a separate running "compensation" variable to track small errors and add them back in the next step.

Log-Sum-Exp

In ML, calculating probabilities often involves softmax: $\frac{e^{x_i}}{\sum e^{x_j}}$.

If $x_i$ is large, $e^{x_i}$ overflows. Trick: Subtract max(x) from all inputs. $$\frac{e^{x_i - M}}{\sum e^{x_j - M}}$$ This keeps numbers in a safe range without changing the ratio.

Bit-Level Analysis

Python Inspecting Float Bits 
import struct
import math

def float_to_binary(num):
    # Pack float into 4 bytes (f) or 8 bytes (d)
    # '!' = Network (Big-Endian), 'f' = float32
    # Here we use 'd' for double (Python float)
    packed = struct.pack('!d', num)
    # Unpack as integers to see bits
    integers = struct.unpack('!Q', packed)[0]
    # Format as 64-bit binary
    return f"{integers:064b}"

# 1. Inspecting 0.1
bits = float_to_binary(0.1)
print(f"0.1 bits: {bits}")
# Sign(1) Exponent(11) Mantissa(52)
s = bits[0]
e = bits[1:12]
m = bits[12:]
print(f"S={s} E={e} M={m[:10]}...")

# 2. Catastrophic Cancellation
a = 1.0 + 1e-15
b = 1.0
print(f"\nanive diff: {a - b}") # Correct-ish

a2 = 1.0 + 1e-16 # Below epsilon for double? Close.
print(f"tiny diff:  {a2 - b}")

# 3. Kahan Summation
def kahan_sum(input_vals):
    sum = 0.0
    c = 0.0 # compensation
    for x in input_vals:
        y = x - c
        t = sum + y
        c = (t - sum) - y
        sum = t
    return sum

vals = [0.1] * 10
print(f"\nKahan Sum: {kahan_sum(vals)}")
print(f"Naive Sum: {sum(vals)}")
