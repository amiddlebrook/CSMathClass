Section 0

Why Proof Writing Matters in CS

In Computer Science, a proof isn't just an academic exercise. It's the ultimate form of Code Review. While tests can only show that your code works for specific inputs, a proof shows it works for all possible inputs.

The Programming-Math Mapping:
  • Definitions = Strictly typed data structures and interfaces.
  • Claims (Theorems) = Function signatures and the "Contract" of what they should return.
  • Proof = The implementation logic that converts the input to the guaranteed output.
üõ°Ô∏è Security Connection:

If you're writing a cryptographic algorithm (like AES) or a consensus mechanism (like Paxos), a "small bug" isn't a minor inconvenience‚Äîit's a system-wide failure. These fields depend entirely on proofs because testing can't cover the infinite space of potential attacks.

Section 1

The Standard Proof Structure

The 4-Step Standard:
  1. Setup (Context): Define the universe of discourse. "Let $n$ be an integer."
  2. Definitions (Unpacking): Convert abstract terms to math. "If $n$ is even, then $n = 2k$."
  3. Transformation (Logic): The meat of the proof. Algebraic or logical steps.
  4. Conclusion (Return): Bind the result back to the claim. "Thus, $P$ is true. ‚àé"
üë®‚Äç‚öñÔ∏è Analogy: The Legal Brief

In a courtroom, you can't just assert someone is guilty. You must provide a chain of custody for evidence. A proof is the "chain of custody" for a mathematical truth. If one link is broken (the "Setup" or "Logic"), the whole case (the Theorem) collapses.

The "Mental Stack Trace"

When you read a proof, you are effectively running a debugger on the author's logic. If you find a step that doesn't follow from the previous one, that's a Logic Bug.

Section 2

Definitions

In CS, we call these Interfaces. If you don't define your types correctly, your proof won't compile.

Standard Mathematical Types

\text{Even: } n \in \mathbb{Z} \text{ is even if } \exists k \in \mathbb{Z} \text{ such that } n = 2k. \\ \text{Odd: } n \in \mathbb{Z} \text{ is odd if } \exists k \in \mathbb{Z} \text{ such that } n = 2k+1. \\ \text{Rational: } q \in \mathbb{Q} \text{ if } \exists a, b \in \mathbb{Z} \text{ such that } q = a/b, b \neq 0.
TypeScript Definitions as Types 
// Existential Definition
type Even = number & { __witness: "2k" };
function isEven(n: number): n is Even {
    return n % 2 === 0; 
}

// Recursive Definition (Inductive)
type NaturalNumber = 0 | { succ: NaturalNumber };
// 0 is a Natural. If n is a Natural, succ(n) is a Natural.
Section 3

Claims & Theorems

Not all truths are created equal. In the world of proofs, we use specific names to denote the "Importance" and "Utility" of a statement.

The Tech Stack of Truth:
  • Axiom: The "Hardware" / "OS Kernal". Fundamental truths we assume without proof (e.g., $a+b = b+a$).
  • Theorem: The "Application". A major, standout result that solves a significant problem.
  • Lemma: The "Utility Function". A small result proven only to help prove a larger Theorem.
  • Corollary: The "Plugin". A result that follows almost immediately from a Theorem with very little extra work.
  • Conjecture: The "Feature Request". A statement we suspect is true but haven't found the code (proof) for yet.
Section 4

Writing the Proof

Good proofs use a mix of formal notation and clear English. Think of it like Self-Documenting Code.

Transition Words (Connecting the Pipes):
  1. "Assume..." (Input)
  2. "By definition..." (Reference)
  3. "It follows that..." (Execution)
  4. "Suppose for a contradiction..." (Try-Catch)
  5. "Consequently..." (Return Value)

Full Example: The Square of an Odd is Odd

\text{Claim: If } n \text{ is odd, then } n^2 \text{ is odd.} \\ \text{Proof: } \\ \text{1. Let } n \text{ be an arbitrary odd integer.} \\ \text{2. By definition, } n = 2k + 1 \text{ for some } k \in \mathbb{Z}. \\ \text{3. Squaring both sides, we get } n^2 = (2k + 1)^2 = 4k^2 + 4k + 1. \\ \text{4. Factoring out a 2, we have } n^2 = 2(2k^2 + 2k) + 1. \\ \text{5. Let } m = 2k^2 + 2k. \text{ Since } \mathbb{Z} \text{ is closed under } + \text{ and } \times, m \in \mathbb{Z}. \\ \text{6. Thus } n^2 = 2m + 1, \text{ which is the definition of an odd number.} \\ \text{7. Therefore, } n^2 \text{ is odd. } \blacksquare
Section 5

Constructive Proofs (The CS Way)

In classical math, you can prove something exists without showing how to find it. In Computer Science, we usually prefer Constructive Proofs‚Äîif you say a solution exists, you must provide the algorithm to build it.

Constructive vs. Non-Constructive

Non-Constructive: "There exists an irrational number $a^b$ that is rational." (Proof uses the Law of Excluded Middle, but doesn't tell you which numbers they are).

Constructive: "To prove an even number exists, notice that $2 \times 1 = 2$. Here is the number: 2."

The Curry-Howard Correspondence

This is one of the deepest secrets in CS: A Proof is a Program, and a Formula is its Type.

Python (Conceptual) Proof as Function 
# Proposition: If A and B are true, then A is true.
# Logic: (A ‚àß B) -> A
# Program:
def proof_and_intro(pair: tuple[A, B]) -> A:
    return pair[0]

# Executing this function with valid inputs is "running the proof".
Section 6

Formal Verification (Hoare Logic)

How do we prove a program is correct? We use Hoare Triples: $\{P\} C \{Q\}$.

  • P: Pre-condition (What must be true before).
  • C: Command (The code).
  • Q: Post-condition (What must be true after).
JavaScript Hoare Triple Example 
// Pre-condition: { x == 5 }
x = x + 1;
// Post-condition: { x == 6 }

// To prove this formally, we show that the logic 
// of the command transforms the pre-state into the post-state.
üöÄ Industrial Success:

The seL4 microkernel is the world's first operating system kernel with a mathematical proof of correctness. It is used in drones and secure systems where a single crash could be fatal.

Recursive Proofs (Functional Programing)

In languages like Haskell or OCaml, we often prove properties of recursive functions using induction on the data structure itself.

Haskell Proof of List Length 
-- Theorem: length (xs ++ ys) = length xs + length ys
-- Proof by Induction on xs:
-- Base Case: xs = []
--   length ([] ++ ys) = length ys
--   length [] + length ys = 0 + length ys = length ys
--   Match!
-- Inductive Step: Assume true for xs, prove for (x:xs)...
Section 7

Hall of Shame (Bad Proofs)

Avoid these common logical felonies.

1. Affirming the Consequent
Error: If P‚ÜíQ, and Q is true, then P is true.
Example: "If I am in swimming pool, I am wet. I am wet. Therefore I am in pool." (No, maybe it's raining).
2. Circular Reasoning (Begging the Question)
Error: Using the thing you are trying to prove as proof.
Example: "God exists because the Bible says so, and the Bible is true because God wrote it."
3. Proof by Example
Error: "Primes are odd. 3 is odd, 5 is odd, 7 is odd. Proved."
Counter-example: 2 is Prime and Even. Boom.

Section 9

Automated Proofs & Testing

In Software Engineering, we don't always write formal proofs. We use Property-Based Testing to brute-force check mathematical properties for thousands of inputs.

PythonHypothesis Testing
# Property-Based Testing (using pseudo-hypothesis library style)
import random

def add(a, b): return a + b

def test_commutativity():
    # Attempt to disprove a + b = b + a
    for _ in range(1000):
        a = random.randint(-1000, 1000)
        b = random.randint(-1000, 1000)
        
        # The Assertion (The Claim)
        assert add(a, b) == add(b, a), f"Failed for {a}, {b}"
        
    print("Commutativity holds for 1000 random cases.")

test_commutativity()
# If this passes, it's not a Proof (universally), 
# but it increases confidence significantly.
Section 7

The Hall of Shame (Antipatterns)

Just as there is "Spaghetti Code," there is "Spaghetti Logic." Avoid these common logical felonies.

1. Proof by Example (The Hasty Generalization)

Error: "2 is prime, 3 is prime, 5 is prime... therefore all integers are prime." (False: 4, 6, 8, etc.)

CS Parallel: Testing a function for $n=1, 2, 3$ and assuming it works for $n=1,000,000$. This is why we need induction or formal metrics.

2. Affirming the Consequent

Error: If $P \to Q$ and $Q$ is true, then $P$ must be true. (False: The "Wet Grass" error).

3. Circular Reasoning (Begging the Question)

Error: Assuming the conclusion in one of your steps. "We know $X$ is true because $Y$ is true, and $Y$ is true because $X$ is true."

4. Hand-Waving (Proof by Intimidation)

Error: Using phrases like "It is obvious that..." or "The reader can easily see..." when the step is actually complex or dubious.

LaTeX Source

\begin{algorithmic}
\STATE $low \gets 0$
\STATE $high \gets n-1$

Why CS loves LaTeX:
  • Version Control: It's plain text, so git diff works perfectly.
  • Separation of Concerns: You focus on content; the compiler focuses on layout.
  • BibTeX: Automated bibliography management.

Section 9

Property-Based Testing

If you can't write a formal proof (which takes a lot of time), the next best thing is Property-Based Testing (using libraries like Hypothesis in Python or QuickCheck in Haskell).

PythonBrute-Force Property Check
# Proposition: reversing a list twice returns the original list.
# Property: reverse(reverse(L)) == L

def test_reverse_property(L):
    assert list(reversed(list(reversed(L)))) == L

# Run for 10,000 random lists!
for i in range(10000):
    test_list = [random.randint(0, 100) for _ in range(random.randint(0, 50))]
    test_reverse_property(test_list)

print("Property holds for 10k random samples.")

The "Mechanical Proof": SMT Solvers (Z3)

Sometimes a proof is too tedious for a human to write or verify. We use SMT Solvers like Microsoft's Z3 to prove the validity of logical statements or find counter-examples automatically.

Python (Z3) Proving Logical Equivalence 
# Proposition: not (A or B) == (not A) and (not B)
from z3 import *

A, B = Bools('A B')
# To prove Formula F is valid, we check if Not(F) is UNSATISFIABLE.
s = Solver()
s.add(Not(Not(Or(A, B)) == And(Not(A), Not(B))))

if s.check() == unsat:
    print("Logic Theorem Proved by Machine!")
else:
    print("Found a counter-example.")
Section 10

The Psychology of Proofs

A proof is not just a bunch of symbols; it's a social contract. A proof is successful only if it convinces a skeptical reader.

Levels of Rigor:
  • Level 0: "Nuh-uh" - Total disbelief. The reader thinks you're lying.
  • Level 1: Verification - The reader checks every line line-by-line. (Like a linter).
  • Level 2: Intuition - The reader understands the strategy and could recreate the proof themselves. (Like a senior dev reading your PR).
üí° Tip: Always state your "Proof Strategy" at the beginning. If the reader knows you're using Contradiction, they won't get confused when you assume the opposite of what you're trying to prove.
