Section 2

Algebra & Symbolic Logic (Days 1-2)

In CS, Algebra isn't just about finding $x$. it's about Symbolic Manipulation‚Äîthe ability to rewrite an expression into a more efficient or readable form without changing its value.

The "Compiler" Analogy

When you simplify $2(x+3) + 4x$ into $6x+6$, you are doing exactly what an Optimizing Compiler does. It takes raw, human-written logic and transforms it into the most compute-efficient equivalent.

Exponent Laws (Complexity):
  • $x^a \cdot x^b = x^{a+b}$ (Memory locality)
  • $(x^a)^b = x^{ab}$ (Nested loops)
  • $x^{-n} = 1/x^n$ (Inverse proportionality)
Logarithms (Search):
  • $\log(ab) = \log a + \log b$
  • $\log(a/b) = \log a - \log b$
  • In CS: $\log n$ is the gold standard for search complexity (Binary Search).

Python Lab: Symbolic Refactoring

PythonSymPy Optimization
from sympy import symbols, simplify, expand

x = symbols('x')
# Original "Messy" Logic
expr = (x + 1)**2 - (x - 1)**2

# Let the "Compiler" simplify it
optimized = simplify(expr)
print(f"Original: (x+1)^2 - (x-1)^2")
print(f"Optimized: {optimized}") # Output: 4x

# This transformation saves 3 additions and 2 multiplications!

Common Pitfalls: Algebra

The "Freshman's Dream": $(x+y)^n = x^n + y^n$. This is NEVER true (except in some fields we won't touch yet). Always expand: $(x+5)^2 = x^2 + 10x + 25$, not $x^2 + 25$. Skipping the middle term is like skipping a guard clause in an if statement.

Section 3

Functions & API Design (Day 3)

A mathematical function $f: A \to B$ is a promise. In software, we call this an Interface or an API Contract.

The "Vending Machine" Analogy

A vending machine is a function. You input a code (Domain), and it outputs a snack (Range).
‚Ä¢ Determinant: A1 always gives Chips. (Not chips or cookies randomly).
‚Ä¢ Injective: Every button gives a unique snack.
‚Ä¢ Surjective: There are no empty coils; every snack is obtainable.

Function Composition as a Pipeline

In Unix, we use the pipe operator | to compose functions. $f(g(x))$ is exactly cat file | grep "error".

JavaScriptFunctional Pipeline
const add1 = x => x + 1;
const square = x => x * x;

// Compose: f(g(x))
const addThenSquare = x => square(add1(x));

console.log(addThenSquare(5)); // (5+1)^2 = 36

Common Pitfalls: Functions

Inverse Confusion: $f^{-1}(x)$ is the inverse function, NOT $1/f(x)$. In CS, an inverse is an "Undo" button (like decryption), whereas $1/f(x)$ is just a reciprocal. Confusing them can lead to critical bugs in data reconstruction.

Section 4

Logic & Boolean Algebra (Day 4)

Mathematical logic is the foundation of digital circuitry. Every CPU is just a massive collection of AND, OR, and NOT gates.

De Morgan's Laws

$\neg(P \lor Q) \iff \neg P \land \neg Q$
$\neg(P \land Q) \iff \neg P \lor \neg Q$

Code Parallel: Refactoring complex if statements.

Logic Gates:
  • AND: All inputs must be high.
  • OR: At least one input high.
  • XOR: Exactly one input high (The "Parity" gate).

Python: Bitwise vs Logical

PythonTruth Table Gen
# Printing a truth table for (P AND Q) OR (NOT R)
print(f"{'P':<5} {'Q':<5} {'R':<5} | Result")
for P in [True, False]:
    for Q in [True, False]:
        for R in [True, False]:
            res = (P and Q) or (not R)
            print(f"{str(P):<5} {str(Q):<5} {str(R):<5} | {res}")

Common Pitfalls: Logic

Affirming the Consequent: Just because $P \to Q$ is true and $Q$ is true, it DOES NOT mean $P$ is true.
Example: "If it rains, the grass is wet." The grass is wet. Does it mean it rained? No! The sprinklers could be on. This is a logic bug that causes countless false positives in software alerting systems.

Section 5

Proof Systems: Human & Machine (Day 5)

A proof is a chain of undeniable truths. In modern CS, we distinguish between Human Proofs (convinces people) and Formal Verification (convinces computers).

The Toolbench of Truth:
  • Direct Proof: Straight-line execution. $A \to B \to C$.
  • Contrapositive: Proving the "Exit Condition". $\neg Q \to \neg P$.
  • Contradiction: The "Try-Catch" of math. Assume error, find impossible state.
  • Induction: The "Recursive Call". Prove $P(0)$ and $P(k) \to P(k+1)$.
üõ†Ô∏è Proof Assistants (Lean 4):

Modern mathematicians use Lean or Coq to write proofs that are compiled and checked by a computer. This eliminates human error. If the code compiles, the theorem is true.

Common Pitfalls: Proofs

Proof by Example: Proving $P(1), P(2),$ and $P(3)$ does NOT prove $P(n)$ for all $n$. This is Inductive Testing, not Mathematical Induction. In CS, this is the difference between a unit test (example) and a formal verification (proof).

Section 6

Lambda Calculus vs. Turing Machines

Computer science was born from two different mathematical models of "Computation." They were proven to be equivalent, but they birthed two different ways of thinking about code.

The Turing Machine (State)

A physical-like machine with a tape and a head.
‚Ä¢ Focus: Mutable state, loops, steps.
‚Ä¢ Legacy: C, Java, Python (Imperative).

Lambda Calculus (Logic)

A mathematical system of pure functions.
‚Ä¢ Focus: Transformation, recursion, immutability.
‚Ä¢ Legacy: Haskell, Lisp, React (Functional).

The "Church-Turing" Thesis

Anything that can be computed can be computed by both models. This is the Universal Law of Computation.

Implement the formula $\sum_{i=1}^{n} i = \frac{n(n+1)}{2}$ in two ways: iteratively and recursively. Verify they match.

JavaScriptVerification
const iterativeSum = n => {
    let s = 0;
    for (let i = 1; i <= n; i++) s += i;
    return s;
};

const formulaSum = n => (n * (n + 1)) / 2;

console.assert(iterativeSum(100) === formulaSum(100), "Logic Error!");

Section 14

The Crisis of Foundations (1900-1930)

Building the foundation of CS wasn't easy. It involved a massive intellectual war between the greatest minds of the 20th century.

David Hilbert

Wanted to prove that ALL of math was consistent, complete, and decidable. He wanted a "Master Algorithm" for truth.

Kurt G√∂del

Destroyed Hilbert's dream with the Incompleteness Theorems. He proved that in any sufficiently complex system, there are true statements that cannot be proven.

Alan Turing & The Halting Problem

Turing took G√∂del's logic and applied it to machines. He proved that you cannot write a program that can tell if any OTHER program will ever stop. This is why we can't have a perfect "infinite loop detector" in our compilers.
