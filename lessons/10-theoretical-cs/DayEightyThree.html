Section 1

The Limit of Testing

Edsger Dijkstra famously said: **"Program testing can be used to show the presence of bugs, but never to show their absence!"**

If your input space is infinite (or even just very large, like a 64-bit integer), you can never test every case. Formal Verification uses mathematical logic to cover every possible state in a single proof.

Section 2

Hoare Logic

Tony Hoare introduced a notation to reason about code logic called the **Hoare Triple**:

$$ \{P\} \ C \ \{Q\} $$
  • $P$: The **Pre-condition** (What must be true before $C$ runs).
  • $C$: The **Command** (The code itself).
  • $Q$: The **Post-condition** (What is guaranteed true after $C$ finishes).

If we can prove that $\{P\} \ C$ always leads to $Q$, we have verified the instruction.

Section 3

Invariants & Variants

To prove a `while` loop, we need a **Loop Invariant**: a statement that is true before, during, and after every iteration. If we find an invariant that stays true and the loop eventually terminates, we have proved the loop's correctness.

Section 4

Model Checking

While Hoare logic is about manually writing proofs for code, **Model Checking** is an automated process. It builds a finite state machine of your program and exhaustively searches for a state where a property (like "no deadlock") is violated.

Critical Systems

Model checking is used at AWS, NASA, and for medical devices. If a bug can kill people or cost billions, you don't "test" it—you verify it with a model checker like TLA+ or Spin.

Section 5

Proof Assistants

The gap between a math proof on paper and a program on a computer is narrowing. Languages like **Coq**, **Isabelle**, and **Lean** allow mathematicians to write proofs that are checked by a computer for 100% accuracy.

Section 6

Python: Design by Contract

While Python isn't a "verified" language, we can use the `icontract` library (or simple asserts) to simulate Hoare logic.

Python Formal Contracts 
def deposit(balance: float, amount: float) -> float:
    # Pre-conditions
    assert amount > 0, "Deposit must be positive"
    assert balance >= 0, "Balance cannot start negative"
    
    old_balance = balance
    new_balance = balance + amount
    
    # Post-condition
    assert new_balance == old_balance + amount, "Math failure"
    assert new_balance > old_balance, "Balance did not increase"
    
    return new_balance

# This style of programming makes it much easier to "verify"
# that our code obeys the rules we defined.
try:
    deposit(100, -50)
except AssertionError as e:
    print(f"Verified Failure: {e}")
